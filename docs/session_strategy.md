# Session Strategy

## Goal

Identify anonymous users without requiring login.

## Recommended Identity Model

Use two separate identifiers:

1. `anonymous_user_id`
2. `session_token`

### `anonymous_user_id`

- generated by the backend as a UUID
- persisted in PostgreSQL
- used as the stable owner id for characters and battle requests

### `session_token`

- generated by the backend as a cryptographically random opaque token
- stored in a secure `HttpOnly` cookie
- mapped in Redis to the `anonymous_user_id`

## Why This Split Is Better

- the browser never needs to know the database primary key format beyond what the API returns
- Redis can expire and refresh sessions without changing entity ownership in PostgreSQL
- the session token can be rotated independently from the anonymous user id

## Recommended Redis Key Shape

Key:

- `anon_session:{session_token}`

Value:

```json
{
  "anonymous_user_id": "uuid",
  "created_at": "timestamp",
  "last_seen_at": "timestamp",
  "session_version": 1
}
```

## Recommended Cookie Settings

- `HttpOnly=true`
- `Secure=true` when HTTPS is enabled
- `SameSite=Lax`
- `Path=/`
- `Max-Age=2592000` for a confirmed 30-day session window

## What Should Not Be The Primary Identifier

- IP address
- User-Agent
- browser fingerprint

These are weak, privacy-sensitive, and unstable. They can be logged as abuse signals, but not used as the main identity key.
